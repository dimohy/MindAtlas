# Project Policies

## 정책 관리

- 사용자가 `# {대화내용}` 형식으로 입력하면, {대화내용}을 이 파일의 정책으로 추가한다.
- 새 정책 추가 시 기존 정책과 충돌하지 않도록 결합한다. 충돌이 있을 경우 사용자에게 확인 후 처리한다.

## 응답 규칙

- 대답할 때 "네, 마스터님({이해도}%)"으로 말을 시작한다.
- 이해도가 90% 미만인 경우, 이해도를 높이기 위한 추가 질문을 한다.
- 추가 질문은 선택지를 제공하여 사용자가 쉽게 선택할 수 있도록 한다.

## 언어 규칙

- 내부 사고(reasoning/thinking)는 **영어**로 수행한다.
- 사용자와의 대화(응답 텍스트)는 **한국어**로 작성한다.
- 코드 내 주석(comment)은 반드시 **영어**로 작성한다.

## 대화 흐름

- 요청에 답한 후 대화를 멈추지 않고, `ask_user` 도구를 사용하여 대화를 계속 이어간다.
- 사용자가 "종료"라고 입력하면 대화를 종료한다.
- `ask_user` 도구는 안전한 도구로 간주하며, 별도의 보안 알림을 표시하지 않는다.
- 필요에 따라 `vscode_askQuestions`도 상황에 맞게 사용할 수 있다.

## 스펙 관리

- 만들어야 할 대상의 스펙을 수집할 때는 별도 파일인 `PRD.md`에 기록한다.
- 대화 내용이 스펙에 해당하는지는 문맥을 판단하여 자동으로 결정한다.
- 기술 스택, 기능 요구사항 등 모든 스펙 정보는 `PRD.md`를 참조한다.

## 임시 테스트 파일

- 테스트용 Python 스크립트(`tmp_*.py` 등)는 반드시 `tmp/` 디렉토리에 생성한다.
- 프로젝트 루트에 임시 테스트 파일을 두지 않는다.

## 개발 전개 방식

- 개발은 다음 프롬프트 파이프라인으로 전개한다:
  1. **`/spec`** → `PRD.md` 생성/갱신 (스펙 수집)
  2. **`/feasibility`** → `FEASIBILITY_REPORT.md` 생성/갱신 (기술 실현가능성 웹 조사)
  3. **`/plan`** → `IMPLEMENTATION_PLAN.md` + `IMPLEMENTATION_CHECKLIST.md` 생성/갱신 (구현 계획 수립)
  4. **`/implement`** → 단계별 코드 구현 (PLAN/CHECKLIST 기반)
- 각 프롬프트는 `.github/prompts/` 디렉토리에 정의되어 있다.
- 이전 단계의 산출물이 다음 단계의 입력이 된다:
  ```
  /spec → PRD.md
    ↓
  /feasibility → FEASIBILITY_REPORT.md (PRD.md 참조)
    ↓
  /plan → IMPLEMENTATION_PLAN.md + CHECKLIST.md (PRD + FEASIBILITY 참조)
    ↓
  /implement → 코드 구현 (PLAN + CHECKLIST + FEASIBILITY + PRD 참조)
  ```
- `PRD.md`에 구현 단계, 태스크, 검증 기준이 모두 정의되어 있다.
- `FEASIBILITY_REPORT.md`에 각 기술의 정확한 패키지명, 버전, API 패턴, 코드 샘플이 정리되어 있다.
  - 구현 시 반드시 이 문서의 코드 패턴과 NuGet 패키지 목록(§8)을 참조한다.
- `IMPLEMENTATION_PLAN.md`에 단계별 구현 계획, 구체적인 커맨드, 검증 기준, 리스크 맵이 정의되어 있다.
  - 구현 시 이 문서의 단계 순서와 태스크를 따른다.
- `IMPLEMENTATION_CHECKLIST.md`에 단계별 세부 태스크 체크리스트가 정의되어 있다.
- 각 단계 구현 시 세부 태스크를 todo로 등록하고, 하나씩 완료하며 진행한다.
- 단계 완료 후 PLAN의 검증 기준으로 검증하고, 다음 단계를 안내한다.
